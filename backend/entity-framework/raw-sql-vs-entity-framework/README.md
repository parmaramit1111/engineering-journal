# When Should You Use Raw SQL Alongside Entity Framework?

**Estimated Reading Time:** 7 Minutes

## Skills Demonstrated

- Entity Framework Core
- SQL Optimization
- Performance Engineering
- Enterprise Reporting
- Database Design
- Software Architecture

---

## Overview

Entity Framework Core is an excellent ORM for building business applications. It improves developer productivity, encourages maintainable code, and works well for most CRUD operations.

However, as enterprise applications evolve, certain reporting scenarios become increasingly difficult to express efficiently using LINQ alone.

This article shares a practical approach to deciding when Entity Framework is sufficient and when carefully written Raw SQL becomes the better engineering choice.

---

## Business Scenario

The project initially started as a Proof of Concept (POC).

As new customer requirements were introduced, the application evolved incrementally. Dashboards became richer, reports became more detailed, and business rules continued to grow.

Eventually, reporting queries required:

- Multiple joins
- Dynamic filters
- Role-based access control
- Aggregations
- Large datasets

While Entity Framework continued to work well for transactional operations, some reporting queries became increasingly difficult to optimize.

---

## Why We Started with Entity Framework

Entity Framework was the natural choice because it provided:

- Faster development
- Repository-based architecture
- Better maintainability
- Strong integration with the existing application framework

For most business operations, it remained the right choice.

---

## What Changed?

As reporting requirements expanded, some LINQ queries became increasingly complex.

Typical symptoms included:

- Long-running queries
- Large SQL generated by Entity Framework
- Increased database CPU usage
- Query timeouts
- Complex execution plans

At this point, the problem was no longer Entity Framework itself—it was the complexity of the reporting logic.

---

## Investigation

Before replacing anything, several optimizations were evaluated:

- `AsNoTracking()`
- Removing unnecessary `Include()`
- Using projections instead of full entities
- Reviewing generated SQL
- Analyzing execution plans
- Profiling slow queries

These improvements helped, but a few reports still required greater control over the generated SQL.

---

## Engineering Decision

Instead of replacing Entity Framework, a hybrid approach was adopted.

### Entity Framework

Used for:

- CRUD operations
- Master data
- Simple searches
- Standard business workflows

### Raw SQL

Used for:

- Complex reporting
- Large aggregations
- Multi-table joins
- Performance-critical dashboards

Raw SQL was stored inside repository classes, parameterized properly, and mapped directly to DTOs using Dapper or `FromSqlRaw()` where appropriate.

This kept the architecture clean while allowing reporting queries to be optimized independently.

---

## Results

The hybrid approach delivered measurable improvements.

- Complex reports improved from approximately **15 seconds to around 2 seconds**
- Database CPU utilization decreased
- Dashboard response times improved
- Reporting logic remained maintainable

Most importantly, the development team continued benefiting from Entity Framework for everyday development while using Raw SQL only where it provided clear value.

---

## Lessons Learned

Entity Framework and Raw SQL should not be viewed as competing technologies.

Each solves a different problem.

> **Use Entity Framework to maximize developer productivity. Use Raw SQL when performance and query control become architectural requirements.**

---

## Best Practices

- Start with Entity Framework for most business applications.
- Measure performance before optimizing.
- Review generated SQL and execution plans.
- Optimize LINQ before switching to Raw SQL.
- Use Raw SQL selectively for performance-critical scenarios.
- Keep SQL parameterized and encapsulated within the data access layer.

---

## Key Takeaways

- Entity Framework is an excellent choice for most application development.
- Raw SQL is valuable for complex reporting and performance-critical queries.
- A hybrid approach provides the best balance between productivity and performance.
- Always choose the right tool based on the problem—not personal preference.
